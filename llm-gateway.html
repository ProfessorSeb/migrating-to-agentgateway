LLM Gateway Migration

Translate your LiteLLM proxy config to AgentGateway OSS. Same OpenAI-compatible API, different (better) config.

Concept Mapping

Here's how LiteLLM concepts translate to AgentGateway:

LiteLLM AgentGateway Notes
model_list llm.models Top-level model definitions
model_name name The name clients send in requests
litellm_params.model params.model No provider prefix needed in AgentGateway
litellm_params.api_key params.apiKey os.environ/VAR becomes "$VAR"
fallbacks groups (in advanced config) Groups tried in order; providers in a group are load-balanced
routing_strategy Multiple providers in a group Load balancing across providers within the same group
master_key jwtAuth / policy JWT-based auth replaces simple API keys
num_retries health.eviction Health-based eviction replaces retry counting

Basic Config: Side by Side

A typical LiteLLM setup with OpenAI and Anthropic models, and its AgentGateway equivalent.

✕ LiteLLM — config.yaml
# LiteLLM Proxy Configuration

model_list:
  - model_name: gpt-4o
    litellm_params:
      model: openai/gpt-4o
      api_key: os.environ/OPENAI_API_KEY

  - model_name: gpt-4o-mini
    litellm_params:
      model: openai/gpt-4o-mini
      api_key: os.environ/OPENAI_API_KEY

  - model_name: claude-sonnet
    litellm_params:
      model: anthropic/claude-sonnet-4-20250514
      api_key: os.environ/ANTHROPIC_API_KEY

  - model_name: claude-haiku
    litellm_params:
      model: anthropic/claude-haiku-4-5-20251001
      api_key: os.environ/ANTHROPIC_API_KEY

general_settings:
  master_key: sk-1234

litellm_settings:
  num_retries: 3
  request_timeout: 30
  fallbacks:
    - claude-sonnet: [gpt-4o]
  drop_params: true

router_settings:
  routing_strategy: simple-shuffle
✓ AgentGateway — config.yaml
# AgentGateway LLM Gateway

llm:
  models:
    - name: gpt-4o
      provider: openAI
      params:
        model: gpt-4o
        apiKey: "$OPENAI_API_KEY"

    - name: gpt-4o-mini
      provider: openAI
      params:
        model: gpt-4o-mini
        apiKey: "$OPENAI_API_KEY"

    - name: claude-sonnet
      provider: anthropic
      params:
        model: claude-sonnet-4-20250514
        apiKey: "$ANTHROPIC_API_KEY"

    - name: claude-haiku
      provider: anthropic
      params:
        model: claude-haiku-4-5-20251001
        apiKey: "$ANTHROPIC_API_KEY"



# No master_key needed - use JWT auth
# No retry config needed - use health eviction
# No routing_strategy needed - use groups
# Env vars use "$VAR" syntax

Key Differences to Note

1. No Provider Prefix in Model Name

LiteLLM uses openai/gpt-4o format. AgentGateway uses a separate provider field and the raw model name.

✕ LiteLLM
model: openai/gpt-4o        # provider/model
model: anthropic/claude-sonnet-4-20250514
✓ AgentGateway
provider: openAI             # separate field
params:
  model: gpt-4o             # just the model

2. Environment Variable Syntax

Different syntax for referencing environment variables.

✕ LiteLLM
api_key: os.environ/OPENAI_API_KEY
✓ AgentGateway
apiKey: "$OPENAI_API_KEY"

3. Fallback / Failover

LiteLLM uses a flat fallbacks list. AgentGateway uses ordered groups for failover with health-based eviction.

✕ LiteLLM
litellm_settings:
  fallbacks:
    - claude-sonnet: [gpt-4o]
    - gpt-4o: [claude-sonnet]
  num_retries: 3
✓ AgentGateway
backends:
- ai:
    groups:
    # Primary: Anthropic
    - providers:
      - name: claude-sonnet
        provider:
          anthropic:
            model: claude-sonnet-4-20250514
        health:
          unhealthyExpression: "response.code >= 500 || response.code == 429"
          eviction:
            duration: 30s
    # Fallback: OpenAI
    - providers:
      - name: gpt-4o
        provider:
          openAI:
            model: gpt-4o
Health-based eviction is smarter than retries AgentGateway temporarily evicts unhealthy providers (e.g., rate limited or 500 errors) for a configurable duration, then re-adds them. No wasted retry cycles.

Bonus: Features You Get for Free

Prompt Enrichment

Automatically prepend or append system prompts to every request. No client changes needed.

policies:
  ai:
    promptEnrichment:
      prepend:
      - role: system
        content: "You are a helpful assistant. Always respond in English."

Prompt Guard / PII Detection

Block or redact sensitive data before it hits the LLM provider.

policies:
  ai:
    promptGuard:
      regex:
        matches:
        - pattern: "\\b\\d{3}-\\d{2}-\\d{4}\\b"    # SSN pattern
          action: REJECT

Built-in Admin UI

AgentGateway ships with a web UI at http://localhost:15000/ui for inspecting config, routes, and backends in real time. No separate dashboard needed.

Migration Steps

Export your LiteLLM config

Copy your current config.yaml and note all models, API keys, and settings.

cat /path/to/litellm/config.yaml

Translate models to AgentGateway format

For each entry in model_list, create an entry under llm.models. Remove the provider prefix from the model name, use the provider field, and switch env var syntax.

Translate fallbacks to groups (if using advanced config)

If you need failover, switch from the simple llm.models shorthand to the full binds/listeners/routes config with groups.

Set up auth (if needed)

Replace master_key with JWT auth or remove auth entirely for dev. See the AgentGateway docs for JWT setup.

Test with the same curl command

The API is identical. Just change the port if needed.

# Works with both LiteLLM and AgentGateway!
curl http://localhost:4000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "messages": [{"role": "user", "content": "Hello!"}]
  }'
