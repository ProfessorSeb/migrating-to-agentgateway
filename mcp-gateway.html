MCP Gateway Migration

Migrate your MCP server configuration from LiteLLM to AgentGateway's native, protocol-aware MCP gateway.

Why AgentGateway for MCP?

LiteLLM added MCP support in v1.80.18. AgentGateway was built MCP-native from day one, with capabilities that go beyond basic proxying.

🔌

Fan-out Aggregation

A single tools/list call fans out to all backend MCP servers and aggregates results. Clients see one unified tool catalog.

🔐

Per-Tool Authorization

CEL-based rules control who can call which tools. 'jwt.sub == "admin" && mcp.tool.name == "deploy"'

🔃

OpenAPI to MCP

Automatically convert any OpenAPI spec into MCP tools. Legacy APIs become agent-callable without writing adapter code.

Concept Mapping

LiteLLM AgentGateway Notes
mcp_servers backends[].mcp.targets MCP servers defined as backend targets
transport: "stdio" stdio: block Same concept, different syntax
url: "https://..." mcp: { host: "https://..." } Remote MCP servers
command + args cmd + args Nearly identical
env env Same
auth_type policies.backendAuth Auth handled as a policy layer
N/A openapi: block AgentGateway can expose OpenAPI specs as MCP tools

Example: Everything MCP Server

The popular @modelcontextprotocol/server-everything test server, configured in both systems.

✕ LiteLLM — config.yaml
model_list:
  - model_name: gpt-4o
    litellm_params:
      model: openai/gpt-4o
      api_key: os.environ/OPENAI_API_KEY

mcp_servers:
  everything_mcp:
    transport: "stdio"
    command: "npx"
    args:
      - "-y"
      - "@modelcontextprotocol/server-everything"

general_settings:
  master_key: sk-1234
  store_model_in_db: true
✓ AgentGateway — config.yaml
binds:
- port: 3000
  listeners:
  - name: mcp-gateway
    protocol: HTTP
    routes:
    - name: mcp-tools
      backends:
      - mcp:
          targets:
          - name: everything
            stdio:
              cmd: npx
              args:
                - "-y"
                - "@modelcontextprotocol/server-everything"

Example: Multiple MCP Servers

A production setup with multiple MCP backends (stdio + remote), auth, and access control.

✕ LiteLLM — config.yaml
mcp_servers:
  # Local stdio server
  everything_mcp:
    transport: "stdio"
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-everything"]

  # GitHub MCP server
  github_mcp:
    transport: "stdio"
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_TOKEN: os.environ/GITHUB_TOKEN

  # Remote MCP server
  remote_mcp:
    url: "https://my-mcp-server.example.com/mcp"
    static_headers:
      X-API-Key: "my-secret-key"
✓ AgentGateway — config.yaml
binds:
- port: 3000
  listeners:
  - name: mcp-gateway
    protocol: HTTP
    routes:
    - name: all-tools
      backends:
      - mcp:
          targets:
          # Local stdio server
          - name: everything
            stdio:
              cmd: npx
              args: ["-y", "@modelcontextprotocol/server-everything"]

          # GitHub MCP server
          - name: github
            stdio:
              cmd: npx
              args: ["-y", "@modelcontextprotocol/server-github"]
              env:
                GITHUB_TOKEN: "$GITHUB_TOKEN"

          # Remote MCP server
          - name: remote
            mcp:
              host: https://my-mcp-server.example.com/mcp

      policies:
        backendAuth:
          passthrough: {}
        mcpAuthorization:
          rules:
          - cel: 'jwt.sub != ""'
Unified tool catalog When an MCP client calls tools/list on AgentGateway, it fans out to ALL backend servers and returns a merged list. Clients see one gateway, not three separate servers.

Bonus: OpenAPI to MCP (AgentGateway Only)

Turn any REST API with an OpenAPI spec into MCP tools automatically. No adapter code needed.

backends:
- mcp:
    targets:
    # Your existing REST API becomes MCP-callable
    - name: petstore
      openapi:
        schema:
          file: ./petstore-openapi.json
        host: petstore.example.com:443
Legacy API migration If your agents currently call REST APIs directly, you can wrap them as MCP tools through AgentGateway without touching the API itself. This enables tool discovery, authorization, and observability.

Full Setup: LLM + MCP in One Config

AgentGateway can run both LLM proxy and MCP gateway on different ports in a single process.

# Combined LLM + MCP Gateway

binds:
# Port 4000: LLM Gateway (OpenAI-compatible API)
- port: 4000
  listeners:
  - name: llm
    protocol: HTTP
    routes:
    - name: models
      backends:
      - ai:
          groups:
          - providers:
            - name: gpt-4o
              provider:
                openAI:
                  model: gpt-4o
                  apiKey: "$OPENAI_API_KEY"
            - name: claude-sonnet
              provider:
                anthropic:
                  model: claude-sonnet-4-20250514
                  apiKey: "$ANTHROPIC_API_KEY"

# Port 3000: MCP Gateway
- port: 3000
  listeners:
  - name: mcp
    protocol: HTTP
    routes:
    - name: tools
      backends:
      - mcp:
          targets:
          - name: everything
            stdio:
              cmd: npx
              args: ["-y", "@modelcontextprotocol/server-everything"]

frontendPolicies:
  tracing:
    host: localhost:4317
  accessLog:
    add:
      backend: backend

Migration Steps

List your MCP servers

Identify all entries in your mcp_servers block. Note which are stdio vs remote, and what auth they use.

Create MCP targets

For each LiteLLM MCP server, create a target under backends[].mcp.targets:

  • transport: "stdio" + command/args → stdio: { cmd, args }
  • url: "https://..." → mcp: { host: "https://..." }
  • env variables: os.environ/VAR → "$VAR"

Configure auth policies

Replace static_headers with backendAuth policies. Add jwtAuth and mcpAuthorization for client-facing auth.

Test with MCP Inspector or Claude Desktop

Point your MCP client to http://localhost:3000 and verify tools/list returns all expected tools.
