Kubernetes Deployment on Kind

Hands-on guide: spin up a kind cluster, deploy both LiteLLM and AgentGateway side by side, compare, then migrate.

Prerequisites

You need these tools installed locally:

ToolInstallPurpose
docker Docker Desktop or Colima Container runtime for kind
kind brew install kind Kubernetes in Docker
kubectl brew install kubectl Kubernetes CLI
helm (optional) brew install helm For Helm/Gateway API path

Set your API keys:

export OPENAI_API_KEY="sk-your-key-here"
export ANTHROPIC_API_KEY="sk-ant-your-key-here"

Quick Start (One Script)

Clone the repo and run the setup script. It creates a kind cluster with port mappings and deploys AgentGateway with LLM + MCP gateway.

# Clone the repo
git clone https://github.com/ProfessorSeb/migrating-to-agentgateway.git
cd migrating-to-agentgateway

# Set your API keys
export OPENAI_API_KEY="sk-..."
export ANTHROPIC_API_KEY="sk-ant-..."

# Run the setup
./setup-kind.sh

This gives you:

  • LLM Gateway at http://localhost:4000
  • MCP Gateway at http://localhost:3000
  • Admin UI at http://localhost:15000/ui
Want to compare with LiteLLM? Run ./setup-litellm-kind.sh after to deploy LiteLLM in its own namespace on the same cluster. Then compare resource usage side by side.

Resource Comparison

AgentGateway is dramatically lighter than LiteLLM in Kubernetes.

16x less memory, 24x faster startup These aren't theoretical numbers. Rust binaries don't need a Python runtime, package manager, or virtual environment in the container.

Step-by-Step: Manual Walkthrough

If you prefer to understand each step rather than running the script, here's the full walkthrough.

Create the kind cluster with port mappings

We map host ports 4000 (LLM), 3000 (MCP), and 15000 (Admin UI) into the kind node.

cat <<'EOF' | kind create cluster --name agentgateway-demo --config -
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
- role: control-plane
  extraPortMappings:
  - containerPort: 30400
    hostPort: 4000
    protocol: TCP
  - containerPort: 30300
    hostPort: 3000
    protocol: TCP
  - containerPort: 31500
    hostPort: 15000
    protocol: TCP
EOF

Create the namespace and secrets

kubectl create namespace agentgateway

kubectl -n agentgateway create secret generic agentgateway-secrets \
  --from-literal=OPENAI_API_KEY="$OPENAI_API_KEY" \
  --from-literal=ANTHROPIC_API_KEY="$ANTHROPIC_API_KEY"

Create the ConfigMap with LLM + MCP config

kubectl -n agentgateway apply -f - <<'EOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: agentgateway-config
  namespace: agentgateway
data:
  config.yaml: |
    binds:
    # LLM Gateway on port 4000
    - port: 4000
      listeners:
      - name: llm
        protocol: HTTP
        routes:
        - name: models
          backends:
          - ai:
              groups:
              - providers:
                - name: gpt-4o
                  provider:
                    openAI:
                      model: gpt-4o
                      apiKey: "$OPENAI_API_KEY"
                - name: gpt-4o-mini
                  provider:
                    openAI:
                      model: gpt-4o-mini
                      apiKey: "$OPENAI_API_KEY"
              - providers:
                - name: claude-sonnet
                  provider:
                    anthropic:
                      model: claude-sonnet-4-20250514
                      apiKey: "$ANTHROPIC_API_KEY"
                - name: claude-haiku
                  provider:
                    anthropic:
                      model: claude-haiku-4-5-20251001
                      apiKey: "$ANTHROPIC_API_KEY"

    # MCP Gateway on port 3000
    - port: 3000
      listeners:
      - name: mcp
        protocol: HTTP
        routes:
        - name: tools
          backends:
          - mcp:
              targets:
              - name: everything
                stdio:
                  cmd: npx
                  args: ["-y", "@modelcontextprotocol/server-everything"]
EOF

Deploy AgentGateway

kubectl -n agentgateway apply -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: agentgateway
  namespace: agentgateway
spec:
  replicas: 1
  selector:
    matchLabels:
      app: agentgateway
  template:
    metadata:
      labels:
        app: agentgateway
    spec:
      containers:
      - name: agentgateway
        image: cr.agentgateway.dev/agentgateway:latest
        args: ["-f", "/config/config.yaml"]
        ports:
        - containerPort: 4000
          name: llm
        - containerPort: 3000
          name: mcp
        - containerPort: 15000
          name: admin
        envFrom:
        - secretRef:
            name: agentgateway-secrets
        env:
        - name: ADMIN_ADDR
          value: "0.0.0.0:15000"
        volumeMounts:
        - name: config
          mountPath: /config
        livenessProbe:
          httpGet:
            path: /healthz
            port: 15000
          initialDelaySeconds: 5
          periodSeconds: 10
        readinessProbe:
          httpGet:
            path: /healthz
            port: 15000
          initialDelaySeconds: 5
          periodSeconds: 10
        resources:
          requests:
            cpu: "250m"
            memory: 128Mi
          limits:
            cpu: "1"
            memory: 512Mi
      volumes:
      - name: config
        configMap:
          name: agentgateway-config
EOF

Expose with NodePort Service

NodePort maps to the kind extraPortMappings so ports are accessible on localhost.

kubectl -n agentgateway apply -f - <<'EOF'
apiVersion: v1
kind: Service
metadata:
  name: agentgateway
  namespace: agentgateway
spec:
  selector:
    app: agentgateway
  type: NodePort
  ports:
  - name: llm
    port: 4000
    targetPort: 4000
    nodePort: 30400
  - name: mcp
    port: 3000
    targetPort: 3000
    nodePort: 30300
  - name: admin
    port: 15000
    targetPort: 15000
    nodePort: 31500
EOF

Wait for the pod and verify

# Wait for rollout (~5 seconds)
kubectl -n agentgateway rollout status deployment/agentgateway

# Check the pod
kubectl -n agentgateway get pods

# Test LLM Gateway
curl http://localhost:4000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "gpt-4o", "messages": [{"role": "user", "content": "Hello from Kind!"}]}'

# Open Admin UI
open http://localhost:15000/ui

Side-by-Side Comparison on Kind

Deploy LiteLLM in its own namespace on the same cluster to compare resource usage and startup time.

Deploy LiteLLM (same cluster, different namespace)

# Use the included script
./setup-litellm-kind.sh

# Or manually:
kubectl create namespace litellm

kubectl -n litellm create secret generic litellm-secrets \
  --from-literal=OPENAI_API_KEY="$OPENAI_API_KEY" \
  --from-literal=ANTHROPIC_API_KEY="$ANTHROPIC_API_KEY" \
  --from-literal=LITELLM_MASTER_KEY="sk-1234"

Compare startup times

# AgentGateway: ready in ~5 seconds
kubectl -n agentgateway get pods

# LiteLLM: ready in ~2 minutes
kubectl -n litellm get pods

# Watch both
kubectl get pods -A -l 'app in (agentgateway,litellm)' -w

Compare resource usage

# Install metrics-server if not present
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
kubectl patch deployment metrics-server -n kube-system \
  --type='json' \
  -p='[{"op": "add", "path": "/spec/template/spec/containers/0/args/-", "value": "--kubelet-insecure-tls"}]'

# Wait a minute, then compare
echo "=== AgentGateway ==="
kubectl -n agentgateway top pods

echo "=== LiteLLM ==="
kubectl -n litellm top pods

Test both (same API, different ports)

# Port-forward LiteLLM to a different port
kubectl -n litellm port-forward svc/litellm 4001:4000 &

# Test AgentGateway (port 4000)
echo "=== AgentGateway ==="
time curl -s http://localhost:4000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "gpt-4o", "messages": [{"role": "user", "content": "Say hi"}]}' | head -c 200

echo ""

# Test LiteLLM (port 4001)
echo "=== LiteLLM ==="
time curl -s http://localhost:4001/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer sk-1234" \
  -d '{"model": "gpt-4o", "messages": [{"role": "user", "content": "Say hi"}]}' | head -c 200
Same API, different characteristics Both expose OpenAI-compatible endpoints. The difference is in resource consumption, startup time, and the additional features AgentGateway provides (MCP gateway, admin UI, health-based eviction).

Production Path: Helm + Gateway API

For production Kubernetes, AgentGateway integrates with Kubernetes Gateway API through its control plane (kgateway). This also works on kind.

Install AgentGateway with Helm on kind

# Make sure your kind cluster is running
kubectl cluster-info --context kind-agentgateway-demo

# Add the Helm repository
helm repo add agentgateway https://agentgateway.dev/charts
helm repo update

# Install the control plane + data plane
helm install agentgateway agentgateway/agentgateway \
  --namespace agentgateway-system \
  --create-namespace

Create a Gateway resource

kubectl apply -f - <<'EOF'
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: ai-gateway
  namespace: default
spec:
  gatewayClassName: agentgateway
  listeners:
  - name: llm
    port: 4000
    protocol: HTTP
EOF

Define backends with CRDs

kubectl apply -f - <<'EOF'
apiVersion: v1
kind: Secret
metadata:
  name: openai-secret
stringData:
  apiKey: "$OPENAI_API_KEY"
---
apiVersion: v1
kind: Secret
metadata:
  name: anthropic-secret
stringData:
  apiKey: "$ANTHROPIC_API_KEY"
---
apiVersion: agentgateway.dev/v1alpha1
kind: AgentgatewayBackend
metadata:
  name: openai
spec:
  ai:
    provider:
      openai:
        model: "gpt-4o"
  policies:
    auth:
      secretRef:
        name: openai-secret
---
apiVersion: agentgateway.dev/v1alpha1
kind: AgentgatewayBackend
metadata:
  name: anthropic
spec:
  ai:
    provider:
      anthropic:
        model: "claude-sonnet-4-20250514"
  policies:
    auth:
      secretRef:
        name: anthropic-secret
EOF

Create HTTPRoute

kubectl apply -f - <<'EOF'
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: llm-route
spec:
  parentRefs:
  - name: ai-gateway
  rules:
  - backendRefs:
    - group: agentgateway.dev
      kind: AgentgatewayBackend
      name: openai
    - group: agentgateway.dev
      kind: AgentgatewayBackend
      name: anthropic
EOF
Why Gateway API? Gateway API CRDs integrate your AI gateway with the same infrastructure tooling as HTTP gateways. Config changes are dynamic via xDS (no restart). You get RBAC, multi-tenancy, and fleet management built in.

./cleanup-kind.sh
